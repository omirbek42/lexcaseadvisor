# PRD: LexCaseAdvisor

**Платформа анализа судебных и арбитражных кейсов для поддержки корпоративных юридических решений**

Версия 1.0 · Статус: концепт / MVP

---

## 1. Резюме

LexCaseAdvisor — инструмент для юридических и комплаенс-подразделений компаний, который позволяет загружать собственную базу судебных и арбитражных кейсов (внутристрановых и международных) и находить среди них дела, похожие по фактическим обстоятельствам на текущую корпоративную ситуацию. Ключевая ценность — поиск **по фактам ситуации**, а не по ключевым словам, с объяснением, почему найденный кейс релевантен.

## 2. Проблема

- Поиск похожих дел вручную занимает много времени и зависит от опыта конкретного юриста.
- Внутренняя база накопленных компанией кейсов используется неэффективно — нет инструмента быстрого поиска по фактам.
- Сложно сопоставлять практику разных юрисдикций (национальные суды vs международный арбитраж) в единой логике.
- Решения принимаются без структурированной оценки рисков на основе прецедентов.

## 3. Цель продукта

Дать пользователю инструмент, который по описанию фактических обстоятельств ситуации находит в базе кейсов наиболее похожие судебные/арбитражные дела и помогает выстроить аргументированную правовую позицию.

## 4. Целевые пользователи

| Роль | Потребность |
|---|---|
| Корпоративный юрист / юридический департамент | Быстро найти прецеденты по конкретному спору, оценить шансы и риски |
| Руководитель юридической функции / GC | Видеть сводную аналитику по практике компании и рынку |
| Внешний консультант / арбитражная практика | Сопоставлять практику разных юрисдикций по одному вопросу |
| Комплаенс / риск-менеджмент | Оценивать юридические риски сделок и решений до их принятия |

## 5. Источник данных (MVP)

Закрытая база кейсов, которую пользователь загружает самостоятельно (файлы PDF/DOCX/TXT). Подключение внешних открытых источников (КАД Арбитр, ICSID, CISG-online и т. п.) — за рамками MVP, см. дорожную карту.

## 6. Объём MVP

### Входит
- Загрузка файлов кейсов пользователем, привязка к рабочему пространству.
- Автоматическое извлечение структурированных метаданных кейса.
- Семантический поиск похожих кейсов по свободному текстовому описанию ситуации.
- Карточка кейса: резюме, метаданные, ссылка на исходный документ.
- Сравнение 2–5 найденных кейсов в табличном виде.
- История запросов пользователя.

### Не входит (следующие фазы)
- Автоматическое подключение внешних баз практики.
- Автогенерация процессуальных документов.
- Прогнозная аналитика «вероятность выигрыша» на основе ML.
- Тонкое разграничение прав доступа (на MVP — упрощённая модель ролей).

## 7. Функциональные требования

### 7.1. Загрузка и управление базой кейсов
- Загрузка файлов (PDF, DOCX, TXT), в т. ч. пакетная.
- Автоизвлечение метаданных с возможностью ручной правки.
- Группировка по папкам/тегам/проектам.
- Редактирование и удаление кейсов.

### 7.2. Поиск похожих прецедентов (ключевая функция)
- Свободное текстовое описание фабулы спора.
- Фильтры: юрисдикция, тип спора, период, орган рассмотрения.
- Ранжированный список результатов с оценкой релевантности и объяснением совпадения.
- Подсветка совпадающих фактов/норм между запросом и найденным кейсом.

### 7.3. Карточка кейса
- Автоматическое резюме дела.
- Полный набор структурированных метаданных (см. п. 8).
- Ссылка/файл с исходным документом.
- Похожие кейсы из той же базы.

### 7.4. Сравнение кейсов
- Выбор 2–5 кейсов.
- Табличное представление: факты, нормы, исход, сходства и различия.

### 7.5. Учётные записи и доступ
- Регистрация/вход, рабочее пространство компании.
- Роли MVP: администратор рабочего пространства и участник.
- Изоляция данных между рабочими пространствами.

## 8. Структурированные поля кейса

- Юрисдикция / орган рассмотрения (суд РФ, арбитражный суд, МКАС, ICC, LCIA, SCC, ICSID и др.)
- Инстанция и дата решения
- Стороны спора (с возможностью анонимизации)
- Предмет и тип спора
- Фактические обстоятельства (фабула) — ключевое поле для поиска
- Применённые правовые нормы
- Правовая позиция и аргументация
- Итоговое решение / исход дела
- Теги и комментарии пользователя

## 9. Архитектура (высокий уровень)

| Слой | Назначение | Вариант технологии |
|---|---|---|
| Frontend | Загрузка кейсов, поиск, карточки, сравнение | React + TypeScript, Tailwind |
| Backend API | Бизнес-логика, авторизация, оркестрация | Node.js (NestJS) / Python (FastAPI) |
| Обработка документов | Парсинг PDF/DOCX, OCR | Apache Tika / pdf-parse / Tesseract |
| Извлечение сущностей | Извлечение фактов, норм, исхода из текста | LLM-агент + промпт-шаблоны |
| Векторизация и поиск | Эмбеддинги и поиск по сходству | pgvector / Qdrant / Weaviate |
| Реляционное хранилище | Метаданные, пользователи, доступы, история | PostgreSQL |
| Файловое хранилище | Исходные документы | S3-совместимое хранилище |

**Поток обработки кейса:** загрузка файла → извлечение текста (+OCR при сканах) → LLM-извлечение структурированных полей → векторизация и сохранение → доступность для поиска.

**Поток поиска:** запрос пользователя → векторизация → гибридный поиск (вектор + фильтры по метаданным) → ранжирование → объяснение релевантности → просмотр/сравнение.

## 10. Нефункциональные требования

| Категория | Требование |
|---|---|
| Конфиденциальность | Изоляция данных между рабочими пространствами; шифрование при хранении и передаче |
| Производительность | Отклик поиска не более 3–5 секунд на базе из нескольких тысяч кейсов |
| Масштабируемость | Рост базы до десятков/сотен тысяч кейсов без переписывания поискового слоя |
| Точность извлечения | Контроль качества автоизвлечения, обязательная ручная корректировка |
| Локализация | Русский и английский языки |
| Аудит | Журналирование загрузок, запросов и изменений метаданных |

## 11. Риски

- Конфиденциальность загружаемых документов (коммерческая/профессиональная тайна) — требуется политика хранения и доступа.
- Неполное автоматическое извлечение фактов — система должна показывать степень уверенности, не выдавая результат как окончательную рекомендацию.
- Продукт не заменяет юридическую консультацию — результаты должны явно маркироваться как вспомогательный аналитический материал.
- Различия терминологии между юрисдикциями требуют отдельных шаблонов извлечения метаданных.

## 12. Дорожная карта

| Фаза | Содержание |
|---|---|
| Фаза 1 — MVP | Загрузка собственной базы, структурирование, поиск, карточка, сравнение |
| Фаза 2 | Подключение открытых источников практики (КАД Арбитр, базы арбитражных институтов, CISG/ICSID) |
| Фаза 3 | Расширенная аналитика: статистика по исходам, динамика практики во времени |
| Фаза 4 | Помощник подготовки правовой позиции: черновик аргументации на основе прецедентов (с проверкой юристом) |
| Фаза 5 | Командная работа: совместные проекты, комментарии, согласование позиций |

## 13. Критерии успеха MVP

- Юрист загружает пакет кейсов и получает структурированные карточки без ручного переноса данных.
- По описанию реальной ситуации система возвращает релевантные кейсы из загруженной базы с понятным объяснением сходства.
- Время поиска подходящего прецедента сокращается по сравнению с ручным просмотром базы.
- Пользователи отмечают, что объяснения релевантности помогают обосновать выбор кейса при подготовке позиции.

## 14. Прототип

Интерактивный UI-прототип (React) реализован в `LexCaseAdvisor.jsx`: поиск по фабуле спора, фильтры по юрисдикции/типу, карточка кейса, сравнение до 3 кейсов, имитация загрузки и обработки нового документа. Поиск в прототипе работает на лексическом сопоставлении — в продакшене заменяется на эмбеддинг-поиск согласно разделу 9.
